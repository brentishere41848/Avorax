# Avorax Security Model

Avorax is a defensive antivirus / anti-malware / endpoint protection product. The security model prioritizes user safety, explicit user control, reversibility, least privilege, auditable decisions, and honest product states.

## Security goals

- Detect known malicious files using local signatures, deterministic rules, heuristics, and trust data.
- Provide useful offline scans without requiring an account, cloud service, or internet access.
- Keep UI responsive and show actionable protection, scan, update, quarantine, and error states.
- Quarantine confirmed threats safely and reversibly.
- Verify update packages before applying them.
- Avoid unsupported claims about kernel, pre-execution, cloud, AI, or ransomware recovery capabilities.

## Non-goals and hard boundaries

- Avorax must not implement malware, credential theft, stealth, evasion, destructive payloads, offensive exploit chains, or hidden persistence.
- Avorax must not disable third-party security tools or Windows protections without explicit user intent and documentation.
- Avorax must not include real malware binaries. Tests must use EICAR, synthetic hashes, harmless fixtures, or temporary-file simulations only.
- Avorax must not claim guaranteed protection, certification, production AI detection, kernel enforcement, or ransomware decryption unless those capabilities are implemented, verified, and documented.
- Avorax must not execute downloaded update code without signature/checksum verification.

## Trust boundaries

### Flutter UI

The UI is not a security boundary. It is a local controller and display surface. Any local process or user can potentially influence UI state. Security decisions must be enforced by the local core, guard service, update service, and OS permissions.

Cloud endpoint, project, and public-client-key configuration values must be typed and bounded before API-client use. Cloud endpoints are limited to HTTP(S) absolute URLs; malformed persisted cloud config fails closed into recovery/default handling instead of being trusted.

Cloud response and exception diagnostics are untrusted remote/runtime text and must be bounded before entering user-visible API failure state.

Manual cloud health checks are optional status evidence, not a dependency for local protection. Flutter must keep cloud checks single-flight; duplicate checks must produce settings warning evidence and visible error state instead of silently returning or implying a fresh cloud result. Settings must disable the cloud check control from the shared public in-flight state as well as the checking status.

Cloud operation names shown in UI or audit failures must use stable product-facing wording rather than internal protocol identifiers where possible, while preserving enough context for support and logs.

Protection-run IDs are untrusted cloud/local action tokens. Before a run ID is stored or reused in heartbeat/end-run URLs, it must be bounded and limited to a conservative token alphabet rather than merely URL-encoded after trimming.

Cloud JSON response bodies must pass explicit byte-size limits before parsing and before full buffering. Flutter cloud health, protection-run, heartbeat, detection-report, and quarantine-metadata responses are collected from streamed response chunks with accumulated byte checks and a read timeout before an `http.Response` body is built for parser use.

Cloud timestamp fields are untrusted response metadata and must be bounded before date parsing.

Outbound cloud detection reports must be count-bounded before serialization and network submission.

The optional backend API must not start with implicit database, Redis, or development API-key defaults. Runtime connection strings are required configuration, development seed keys are created only when explicitly enabled, and Redis, `.env`, and migration directory failures must be reported as startup errors rather than silently disabling backing services or skipping migration files.

The optional backend API must not use permissive browser CORS by default. Native Avorax clients do not need browser CORS; any browser-facing UI must define explicit allowed origins and be tested with preflight requests before deployment.

The optional backend API treats cloud request bodies as untrusted. Body size, event batch count, nested JSON size, project/device/session text, SHA-256 fields, quarantine IDs, and detection metadata must be bounded before database writes. The API must keep the Flutter client route contract (`/v1/protection-runs` and `protection_run_id`) wired, and compatibility aliases must not bypass validation.

Backend bearer tokens are untrusted header input. Tokens must be non-empty, bounded, and limited to the documented key alphabet before hashing or database lookup, so malformed or oversized authorization headers cannot drive unbounded auth work.

Backend project identifiers in request bodies are untrusted routing hints. A supplied project UUID must match the authenticated project exactly, and a supplied slug must resolve to that authenticated project row before any device, session, detection, or quarantine write uses it.

Backend route compatibility is part of the protection workflow contract. Dynamic Axum routes must use the supported current router syntax so cloud protection-run, heartbeat, detection, and quarantine flows do not fail at API startup.

Backend project/key provisioning must not be unauthenticated. Until a real admin workflow exists, public project creation must fail closed; local development seeding is an explicit startup mode, not an internet-facing registration path.

Backend API error responses must not leak database, SQL, Redis, filesystem, or infrastructure internals. Internal and database failures should be logged server-side when logging is configured, but client JSON responses use stable generic error text.

Local event history records are untrusted persisted data. Event string fields and timestamp text must be typed and bounded before history/log UI use, and event category/severity values must be canonicalized to supported UI values.

New local event writes treat category and severity as audit classifications, not decoration. The app-side repository rejects unsupported write-time category/severity values before storage instead of silently rewriting them to `app` or `info`; legacy persisted reads may still use conservative defaults so old or forged local history does not block startup.

App-side local event repository migration/loading also treats timestamp text as untrusted and bounds it before date parsing; malformed event rows are dropped without clearing valid neighboring history.

Persisted local event history has a size boundary before JSON parsing. Flutter rejects and removes oversized stored local-event blobs before `jsonDecode`, so corrupt SharedPreferences data cannot force unbounded audit-history parsing during startup, export, or settings/logging flows.

Persisted local event history recovery must preserve bounded failure context. Oversized, wrong-shape, and JSON-decode failures are safe to recover from, but the recovery warning must include a bounded reason so corrupt local audit evidence is visible instead of being silently reduced to a generic reset.

Loaded local event history is retention bounded. The repository applies the same 200-row newest-first cap when loading persisted history as it does when writing new events, so legacy or forged stored audit rows cannot expand log UI or export work beyond the documented local retention window. Checkpoint 1328 source-verifies persisted history with excessive row counts is rejected before per-row parsing, so forged small-record histories cannot spend unbounded CPU/RAM before the retention cap applies.

Local event export failure diagnostics are bounded. If export fails and temporary export cleanup also fails, both exception strings must be trimmed to the local event detail limit before they enter UI error state.

App-side runtime diagnostics from app detection, platform probes, config recovery, local-event maintenance, Settings log export, Device provider failures, and shell local-event notifications are untrusted text. These paths must bound exception strings before they enter visible UI state, snackbar text, recovery reasons, local audit history, or notification summaries, and shared diagnostic helpers must control/NUL-normalize text before applying visible caps. Checkpoint 1315 source-verifies the feature-level Settings and Device diagnostic helpers apply that same boundary before export-log or platform-provider errors are shown. Checkpoint 1316 source-verifies shell notification summaries control/NUL-normalize event message/detail text before display.

Windows platform probes must not rely on ambient `PATH` lookup or raw inline PowerShell command text. Flutter must launch a checked local WindowsPowerShell executable path, encode probe scripts with UTF-16LE `-EncodedCommand`, bound stdout/stderr, and fail visibly when the PowerShell executable is absent, linked, remote, or otherwise ambiguous. Flutter Windows helper roots derived from `SystemRoot` or `WINDIR` must reject NUL and parent-traversal text before any command path is constructed; checkpoint 1305 source-verifies this for platform probes, protected-app process enumeration, update launchers, Explorer report opening, and elevation PowerShell launchers.

Flutter platform-info fallback text is device evidence, not raw environment output. `USER`, `USERNAME`, `USERDOMAIN`, and `PROCESSOR_ARCHITECTURE` must be trimmed, bounded, and control/NUL-normalized before they populate user, permissions, or architecture fields; checkpoint 1308 source-verifies this fallback path.

Platform PowerShell probe output is untrusted subprocess text. The app must collect stdout/stderr through bounded streams, kill timed-out probes, and reject oversized successful output before JSON parsing. Checkpoint 1292 source-verifies that timed-out platform probes report whether the termination request was accepted or failed.

Protected-app process-list output is untrusted subprocess text. App detection must not use unbounded `Process.run` collection for `tasklist` or `ps`; process-list stdout/stderr must be bounded and oversized output must fail visibly. Checkpoint 1292 source-verifies that timed-out process-list commands report whether the termination request was accepted or failed.

Protected-app app-detection diagnostics are also untrusted text. Process-list output, process-launch errors, and install-path probe failures must be bounded and control/NUL-normalized before they reach UI error state or audit evidence; checkpoint 1309 source-verifies this diagnostic boundary.

When UI/runtime diagnostics append an ellipsis, the ellipsis is part of the bound. Truncation helpers must keep the final displayed string within the configured maximum length rather than returning maximum-length content plus extra suffix characters.

Updater subprocess diagnostic truncation markers are also part of the bound. stdout/stderr collection must not append marker text after reaching its configured limit.

Updater subprocess diagnostics must remain bounded after stdout/stderr labels are joined into an install/rollback failure message.

Updater subprocesses must have a finite runtime bound. Timed-out update helpers are killed and reported with bounded diagnostics; the UI must not wait indefinitely for updater child processes.

### Local core IPC

The Flutter app communicates with the local core over stdin/stdout JSON commands. This channel is local and not network-exposed, but requests must still treat paths and JSON fields as untrusted input.

Requirements:

- Validate command names and required fields.
- Checkpoint 911 treats local-core command JSON as an explicit IPC schema. Unknown command fields are rejected during deserialization instead of being ignored, so callers cannot send dead controls and receive a response that appears to honor them.
- Reject oversized command JSON before parsing, using a bounded line reader before allocating the full stdin command.
- Canonicalize and validate paths before scanning, allowlisting, quarantining, restoring, or deleting. Local-core IPC path fields reject empty, NUL-containing, oversized, or excessive path-list payloads before scan/watch/action handlers use them.
- Bound local-core IPC labels, notes, IDs, mode strings, threat names, engine names, and ransomware-guard path lists before they influence parsing, persistence, quarantine metadata, training labels, restore/delete actions, or config writes.
- Bound local-core response diagnostics before they enter action results, watcher state, UI error text, or event history. Malformed action or watcher protocol responses must not carry unbounded raw `error` strings into the controller.
- `watch_poll_scan` IPC is finite user-mode polling, not persistent service monitoring. Requests must keep duration, interval, event count, file count, and depth bounded; responses must expose poll metrics, limitations, and scan errors; and app-side callers must surface failures instead of silently treating a missed poll as protection success.
- Bound local-core list-response diagnostics before they enter quarantine or allowlist refresh failures.
- Decode local-core stdout IPC with malformed-tolerant UTF-8 before line splitting so invalid bytes are routed to bounded protocol diagnostics rather than strict decoder exceptions.
- Bound individual local-core stdout IPC lines before JSON parsing so one oversized line cannot force unbounded buffering in the Flutter client.
- Bound local subprocess diagnostics before they enter local-core UI recovery paths. Guard self-test stdout/stderr, scan IPC stderr, and elevated PowerShell stderr/launch failures must use bounded diagnostic collection instead of raw stream joins.
- Kill timed-out Guard self-test subprocesses and report bounded timeout diagnostics.
- Kill timed-out scan-cancel IPC subprocesses and report bounded timeout diagnostics.
- Timed-out local-core scan IPC subprocesses must report whether child-process termination was requested or failed, so UI timeout evidence does not imply cleanup succeeded when the kill request failed; checkpoint 1290 source-verifies this evidence in the Flutter IPC timeout path.
- Timed-out Guard self-test IPC, cancel IPC, and elevated PowerShell helper subprocesses must also report whether child-process termination was requested or failed; checkpoint 1292 source-verifies those Flutter helper timeout paths.
- Bound both stdout and stderr from elevated PowerShell helper subprocesses and kill timed-out launchers.
- Bound LocalCoreClient exception diagnostics before they enter local-core UI recovery paths. Self-test parsing, install-report launch, IPC call, cancel IPC, and executable probe exceptions must pass through the shared IPC diagnostic limiter before the controller sees them.
- Client-side local-core and guard-service executable discovery must inspect candidates without following links and must surface filesystem inspection failures as diagnostics rather than silently treating an uninspectable path as missing. Environment overrides must be non-empty after trimming, and explicit overrides should remain the reported path when they fail validation.
- Return structured errors instead of panics.
- Emit structured progress serialization errors instead of blank progress lines if local-core progress JSON serialization fails.
- Avoid logging sensitive file contents.

### Local data migration

Legacy-data migration is a local filesystem copy operation and must treat existing source and destination paths as untrusted.

Requirements:

- Copy only explicit legacy data groups.
- Reject symlinks, junctions, and reparse points in source and destination paths.
- Bound recursion depth.
- Preserve current Avorax destination files rather than overwriting them.
- Stage migration marker and event-log writes before activation.

### Native engine assets

Signatures, rules, trust stores, and ML assets influence verdicts and actions. Engine assets are resolved from installed paths, environment overrides, or repository assets during development.

Requirements:

- Installed-layout asset alias selection must use non-following metadata inspection. If a preferred Avorax-named asset path is present, broken, linked, reparse-backed, or uninspectable, selection must not silently fall back to a legacy Zentor-named alias; the downstream loader should fail visibly with path context.
- Installed-layout asset alias candidate lists must not be empty. Missing candidates may still fall back to the first declared candidate for visible downstream loader errors, but an empty candidate list is a programmer/configuration invariant failure and must not silently become an empty/default path.
- Installed-layout first-candidate fallback must be a named branch. If no candidate is present or uninspectable, selecting the first declared candidate is compatibility behavior for downstream error context, not an implicit optional default.
- The installed native `engine` asset root must itself be a regular directory inspected without following links. A symlinked or reparse-backed engine root is not trusted as an installed asset directory, because per-file checks cannot prove parent-directory containment through a redirected root.
- If the installed native `engine` asset root cannot be inspected for a reason other than `NotFound`, native config selection must not silently fall back to development assets. It should select installed-layout asset paths so downstream pack/model/trust loaders report the installed-root failure with path context.
- Local-core engine asset discovery must use the same non-following marker-directory rule for installed `engine` roots and development `assets\zentor_native` roots before surfacing health/install-path evidence or passing the root to the native engine.
- Local-core engine asset discovery must report current-directory failures instead of silently substituting `"."`; health responses must surface native-engine locator failures as error evidence rather than fake install paths.
- Local-core engine asset candidate normalization must keep compatibility branches visible. Environment-supplied `engine` directories use explicit parent/no-parent handling, and failed candidate canonicalization may keep the original candidate only through a named fallback before non-following marker checks.
- Native engine default configuration must not silently substitute `"."` when the current working directory cannot be read. Runtime flows should pass explicit asset roots; default discovery must use `try_default() -> Result<Self>` and return contextual current-directory errors instead of exposing a panicking production `Default` implementation.
- Native engine environment roots are native trust boundaries. `AVORAX_ENGINE_DIR`, `AVORAX_ENGINE_ROOT`, native quarantine overrides, ProgramData-derived roots, and HOME-derived roots must reject NUL and parent-traversal segments before `PathBuf` construction and absolute-local path validation. Checkpoint 1209 source-verifies this for native engine config parsing.
- Native engine installed-asset alias selection must make the required first candidate explicit in the function boundary. Optional alias candidates may be scanned after it, but an empty candidate list must not be represented by a runtime `expect` panic or a hidden fallback path.
- Validate pack metadata and hashes where supported.
- Reject unsafe broad signatures/rules in compiler/gates.
- Inspect native signature and rule pack paths without following filesystem links before primary load, sibling enumeration, or bounded reads; present symlinks, Windows reparse points, directories, and other non-regular pack paths must fail visibly instead of being followed, hidden as missing, or silently skipped.
- Native signature compiler source and output paths are definition-chain evidence. As of checkpoint 858, compiler input JSON must be inspected as a regular non-link/non-reparse file before bounded reads, and compiled pack/metadata outputs must use checked output directories, existing-target preflight, exclusive UUID temporary files, sync, cleanup-on-failure, and rename activation rather than direct final writes. Relative output paths must be resolved through a fallible checked current-directory step before temp/preflight/activation logic, not through a hidden `Path::new(".")` parent fallback. Checkpoint 1279 source-verifies that compiler source JSON reads carry the inspected metadata into the read path and enforce the byte cap against actual bytes read before UTF-8 and JSON parsing.
- Native signature packs are detection inputs, not extensible policy blobs. `SignaturePack`, `SignaturePackMetadata`, `NativeSignature`, and `SignatureMatch` schemas must reject unknown fields so enabled/allow overrides or policy-looking controls cannot be silently ignored while appearing present in local signature evidence.
- Size-bound native ML `.zmodel` files before JSON parsing.
- Inspect native ML `.zmodel` paths without following filesystem links before discovery or bounded reads; present symlinks, Windows reparse points, directories, and other non-regular model paths must fail visibly instead of being followed or treated as missing.
- Treat development ML assets as development-only unless metadata says `production_ready=true` and release gates approve.
- Static ML release metadata is a trust boundary. `ml/evaluate_model.py` must read a bounded regular JSON metadata file, require typed `production_ready`, finite unit `false_positive_rate`/precision/recall metrics, positive validation sample counts, and ordered thresholds; development overrides may allow a non-production build to continue, but the report must still mark the model as not release-valid.
- Static ONNX export must not synthesize production-ready metadata. `ml/export_onnx.py --production-ready` requires explicit dataset name, positive training and validation sample counts, acceptable FPR/precision/recall evidence, and checked output paths before model metadata can claim production readiness.
- Static ML feature-build and training-summary tooling must treat local JSON/JSONL as untrusted input. `ml/build_features.py` and `ml/train_model.py` must bounded-read regular non-linked files, validate schema-owned feature names and values, reject unsupported labels or empty datasets, write outputs atomically to regular targets, and keep `train_model.py` output explicitly development-only until a separate release export/evaluation path supplies production evidence.
- Offline Python ML tooling dependencies are development-only and must be pinned to exact versions. Release branches must not use weekly/nightly ML packages or unpinned Python dependencies for model export or schema validation, and a release build host must verify package licenses and imports before production model promotion.
- Release dependency review must include generated package-manager lockfiles. Root workspace and update-service Rust lockfiles must be generated with Cargo before release/SBOM review; lockfile contents must not be handwritten or inferred from manifests.
- No-malware-binaries release gates are launch and filesystem trust boundaries. PowerShell and shell gate wrappers must use an explicit checked Python executable instead of ambient `PATH` and run the verifier without writing Python bytecode; PowerShell must restore any caller bytecode-generation environment after execution and must not silently swallow restore failures. The shell wrapper must validate the repository root, Python executable, and verifier script as absolute non-linked paths before launch and reject symlink path components. ### Guard service and driver IPC

Pre-execution and real-time blocking are high-trust paths. The user-mode guard service can provide best-effort monitoring and post-launch action. True pre-execution blocking requires the signed Windows driver path.

Local-core `start_watch` is watch-root planning evidence unless an installed long-running watcher/service report proves more. One-shot local-core IPC may validate accessible directory roots and return best-effort watch-plan state, but it must report limitations such as one-shot planning, no persistent service monitor, and no kernel pre-execution blocking. UI copy must describe these roots as prepared/limited, not as a verified persistent realtime monitor.

Local-core `watch_poll_scan` is finite user-mode polling evidence, not installed realtime service evidence. It may validate accessible watch roots, establish a baseline, poll for stable new or changed regular files within strict duration/interval/max-event/file-count/depth limits, and scan/quarantine confirmed detections after writes are observable. It must report limitations such as finite polling, post-write detection only, bounded polling limits, no persistent service monitor, and no kernel pre-execution blocking. Release-binary watch-poll smoke evidence may prove one bounded command session observes and handles a safe fixture, but it must not be used to claim background monitoring after the command exits, OS notification coverage, driver enforcement, installed UI/service E2E, or pre-execution blocking.

Driver status labels are pre-execution trust evidence. Home, Protection, Device, and Settings must distinguish stopped, installed, missing, unknown, blocked, and unavailable driver states instead of collapsing every non-running status into missing; only a confirmed `missing` status may be labeled missing.

Driver status defaults and IPC parser fallbacks must stay conservative. Until local-core health provides a valid `driver_status`, Flutter state and health parsing use `unknown` rather than `missing`; the `missing` label is reserved for an explicit local-core `driver_status: "missing"` value.

Guard status labels are protection-readiness evidence. Home, Protection, Device, and Settings must distinguish `off`, `missing`, and `unknown` Guard states instead of using `off` or not-running as the default fallback for malformed or unsupported labels.

Guard status defaults and IPC parser fallbacks must stay conservative. Until local-core health provides a valid `guard_status`, Flutter state and health parsing use `unknown` rather than `off`; the `off` label is reserved for an explicit Guard Service status report.

Native ML status labels must match the local-core health status set. UI surfaces use explicit labels for `loaded`, `developmentModel`, `modelMissing`, and `error`; stale aliases such as `active` must not make a loaded model appear unavailable or an error appear missing.

Native Engine UI labels must match the local-core health status set. `unavailable` is not an engine `error`; Home, Protection, Device, and Settings surfaces must distinguish `ready`, `error`, unavailable, and unknown states.

AI model metadata is release/readiness evidence. Missing or malformed health IPC for `ai_model.feature_schema_version` must render as unavailable, not as a default schema version such as `1.0.0`; only actual model metadata may provide a concrete schema version. Settings must display this metadata from health state instead of hardcoding a schema claim.

Scan diagnostics Core Service labels must match the local-core health status set. The Engine unavailable panel distinguishes `running`, `stopped`, `missing`, `installed`, `unknown`, `unsupported`, `error`, and unavailable fallback states; `unsupported` is valid only when the local core reports that Windows service status probing is not available on the current OS.

Protection checklist Core Service labels must also match the local-core health status set. The Protection readiness checklist distinguishes `running`, `stopped`, `missing`, `installed`, `unknown`, `unsupported`, `error`, and unavailable fallback states instead of reducing local-core service errors or unsupported-platform service probes to generic unknown evidence.

Local-core Core Service unsupported status is parser-visible. The non-Windows local-core status producer emits `unsupported`; Flutter health parsing must accept that value and Scan/Protection labels must render it explicitly instead of treating it as malformed IPC or an unavailable fallback.

### Update packages

`.aup` update packages are untrusted until fully verified.

Requirements:

- Verify product identity, format version, target version/channel, manifest signature, and per-file payload hashes. Installed-version discovery failures are failed update checks with explicit context; the build-config version may label the failed result but must not drive update comparison as verified installed state.

Accepted true/false values are narrowly enumerated; malformed, empty, or non-Unicode values are configuration errors that stop CLI verify/apply policy construction instead of silently defaulting to production/false.

Empty, NUL-containing, non-Unicode, malformed, or oversized configured key values are configuration errors, not unknown-key or no-key defaults.
- Update key-generation helpers must not ignore arguments while producing signing material. `avorax_generate_update_key` generates only with no arguments, supports explicit help without creating key material, and rejects unexpected or trailing arguments visibly so output-path or mode typos cannot leak a private seed to stdout under a false assumption.
- Stage updates before applying.
- Roll back on failure where possible. If payload copying fails after a rollback snapshot exists, the update service now attempts to restore that snapshot, restart services, and write a structured failed update report containing rollback status.
- Do not execute downloaded installers or scripts as a normal in-app update path.

Local-core native ML health status must not default a missing native model version to an empty non-development version. If the native engine reports a loaded ML model without version evidence, local-core health reports an error-like native ML status instead of `loaded`.

Local filename-derived AI/static and heuristic signals must preserve evidence when a path has no usable leaf name. Missing or blank `file_name()` values use display-path fallback, with an explicit unknown-path marker only when even display text is empty, instead of silently turning filename risk, double-extension checks, or heuristic executable-name checks into empty-string signals.

Local-core scan error lists are untrusted IPC evidence. `scanErrors` and `scan_errors` must be parsed as bounded non-empty diagnostic strings, with explicit diagnostics for malformed lists, truncated lists, or malformed entries instead of arbitrary stringification.

Local-core threat timestamps are untrusted IPC evidence. Malformed `detectedAt`/`detected_at` fields must add scan errors instead of silently inventing the current time as detection evidence.

Local-core threat lists are untrusted IPC evidence. Non-object threat rows must add indexed scan errors instead of being silently filtered from the report.

Local-core risk-score evidence is untrusted IPC evidence. Malformed risk score objects, scores, reason lists, and engine lists must add scan errors instead of silently defaulting to low-risk or filtering explainability evidence.

Quarantine and allowlist records are action-target IPC evidence. Malformed list rows must fail the list request with indexed diagnostics instead of being silently dropped from the UI.

Protection checklist quarantine readiness must not be a hardcoded success claim. Without a dedicated quarantine-health field, the checklist may show availability only when local engine evidence is ready and otherwise must show unknown rather than `Ready`.

Quarantine and allowlist record fields must not silently default supplied malformed evidence. Invalid action-target timestamps, statuses/types, hashes, and booleans invalidate the row so the list request surfaces an indexed parser error.

Local quarantine metadata authentication is a Recovery Vault trust boundary. Rule-file reads in both Guard Service and driver IPC retain non-following regular-file metadata and enforce metadata-size plus actual-read byte limits before parsing. Guard Service and driver IPC sample reads use explicit chunked prefix limits as of checkpoint 1283. Once a rule header has been accepted, malformed `$string` declarations or unquoted pattern values must fail visibly in both Guard Service and driver IPC compatibility parsers instead of being skipped as if the rule simply did not match. `$string` declarations before an accepted rule header remain ignored as out-of-rule text.

Guard compatibility YARA confidence metadata is verdict policy input. Unknown or malformed confidence labels inside an accepted rule must fail visibly instead of silently degrading to low-confidence evidence.

User-mode realtime monitoring limitations are product evidence. If the local core reports watcher limitation tokens, the UI must preserve and display them instead of claiming stronger monitoring than the user-mode watcher can provide. Malformed watcher `active`, `watched_paths`, and `limitations` IPC fields must produce bounded diagnostics instead of silently becoming a clean inactive watcher or empty lists.

Background refresh tasks must not rely on unhandled exceptions for control flow. Protected-app detection and native-engine health refresh failures are user-visible state transitions and audit-log events, not silent task failures or stale scanning/checking states.

Protected-app running-process detection must not convert failed process enumeration into an empty "no running app" signal. Failed `tasklist` or `ps` probes are detection failures with contextual error text so the UI can surface uncertainty instead of implying no supported app is running.

Protected-app process enumeration must not rely on ambient `PATH` lookup for OS tools. Flutter app detection must resolve Windows process enumeration to a checked System32 `tasklist.exe` path and POSIX enumeration to checked `/bin/ps` or `/usr/bin/ps`, using non-following regular-file probes before launch and failing visibly if those commands are absent or ambiguous.

Protected-app known-install-path detection must not convert filesystem probe failures into empty "not found" evidence. Failed known-root, candidate-directory, or executable probes are detection failures with contextual error text so the UI can surface uncertainty and audit `app_detection_failed`.

Manual protected-app selection is not a security boundary, but it controls scan/watch scope and user trust decisions. File/folder picker failures must be visible and audited, and selected-app UI text must remain readable rather than displaying encoding artifacts.

Custom scan target selection is a user-visible security workflow. File/folder picker failures must be represented as failed scan state and audit-log events instead of uncaught button-handler exceptions.

Quick/Full Scan target planning must be best-effort without hiding coverage uncertainty. True missing candidate roots and high-risk folders can be skipped, but filesystem probe exceptions must keep the candidate for local-core path validation so scan status and audit reporting can record explicit skipped/error coverage.

Scan orchestration must always settle into an explicit terminal state. Unexpected local-core scan-call exceptions are failed scans with scan/error audit evidence, skipped target counts, and visible errors, not uncaught task failures or indefinite running UI state.

Protection controls must always settle into explicit UI states. Start, stop, and self-test failures are audited protection events with visible errors, not uncaught task failures or stale starting, stopping, or loading indicators. Protection start/stop must be single-flight at the controller boundary, and Home/Protection start-stop controls must expose that boundary by disabling while a protection action is in flight. Protection self-test must be single-flight at the controller boundary and must not overlap start/stop protection work, because self-test evidence is not trustworthy while Guard policy or watcher state is changing. The Protection and Settings self-test buttons must expose this boundary by disabling while a protection action or self-test is in flight, not only while the generic loading flag is set.

App-side filesystem probes that influence hashing, watcher setup, or scan dispatch must not throw from `existsSync` checks. For scan dispatch, uninspectable candidates are retained for local-core validation/reporting; for other workflows, inaccessible or malformed paths are routed through the workflow's visible error/reporting path.

Protection watcher setup follows the same uncertainty rule for configured directory scopes. If Flutter cannot inspect a configured watcher candidate, it keeps the candidate for local-core watcher validation so the watched scope is not silently reduced before the service can report the path problem.

Persisted UI configuration is untrusted local input. Malformed, non-object, or schema-invalid config records must not crash startup or preserve partial policy state; the UI falls back to build defaults unless a valid developer override is present.

Protection settings writes must be honest. If persistence or local-core propagation fails while changing protection mode or ransomware-guard policy, the UI logs an error event and shows a visible failure instead of claiming the policy was saved.

Turning protection off requires the explicit stop flow. The protection-profile setter must reject `ProtectionMode.off`, emit a failed protection-mode event, and show a visible error; only `stopProtection()` may configure guard mode `off` while also disabling realtime preference, ending cloud lifecycle state, stopping user-mode watchers, and clearing protection runtime status.

If startup or manual config editing leaves `ProtectionMode.off` in local config, `startProtection()` resets the profile to Balanced, persists that recovery, logs a warning event, and sends the recovered profile to local-core instead of presenting active protection with guard mode disabled.

Scan startup must report native-engine initialization failures as engine-unavailable scan reports. If the engine cannot initialize, planned files are counted as skipped, no file is reported scanned, and the report includes an explicit scan error rather than returning a generic command failure or a clean result.

Auto-actions must never be silent. If a scan detects a threat but automatic quarantine fails, the detection remains visible, the file is not counted as quarantined, and the report includes an explicit auto-quarantine scan error.

Allowlist data is trust data. Environment-backed allowlist loading must fail visibly on read or parse errors. `ZENTOR_ALLOWLIST_FILE` must resolve from absolute, local, NUL-free, parent-traversal-free text before any allowlist file is read or written. During scans, unavailable allowlist data must not suppress detections; Avorax records a scan error and treats the file as not allowlisted.

Allowlist hits must remain explainable detection evidence, not silent clean results. A confirmed detection that matches an active user allowlist entry is reported as `allowlisted` with an allowlist recommendation, must not be auto-quarantined in confirmed-only mode, must not receive quarantine IDs or payload paths, and must leave the source file unchanged. Checkpoint 2111 verifies this with the release local-core binary using a benign exact-hash fixture and isolated process-local trust/quarantine roots.

The UI must preserve that trust-data distinction. A failed allowlist refresh is displayed as an error and must not be rendered as an empty allowlist.

Quarantine records are security evidence. A failed quarantine refresh is displayed as an error and must not be rendered as an empty quarantine list.

Protection policy writes must preserve local-core error details. UI guard-mode and ransomware-guard configuration failures display the exact local-core error instead of a generic boolean failure.

Real-time watcher state must not fake a clean off state on malformed or failed local-core responses. Watcher start/stop failures preserve error text and are surfaced in protection UI warnings.

Scan reports with file errors must remain visible after completion. The UI keeps a global warning when `scanErrors` is non-empty, even if detections were also found, because skipped or failed files were not reported clean. Scan completion audit events with skipped/error coverage must use scan warning severity rather than ordinary info completion.

Scan report parsing must preserve Rust local-core wire fields. Flutter accepts both camelCase and legacy snake_case for scan reports and progress so error lists and counters are not silently lost during IPC compatibility transitions.

Threat result parsing must preserve detection evidence. Flutter accepts both camelCase and snake_case for threat metadata and nested risk-score action/engine fields so detections are not silently downgraded to unknown categories or review actions.

Quarantine and allowlist parsing must preserve evidence and trust history. Flutter accepts both camelCase and snake_case for record IDs, paths, timestamps, actions, and authors so evidence-store compatibility drift does not silently render blank/default records.

### Full Scan

Full Scan may traverse drives or home filesystem areas but must avoid unsafe traversal:

- Do not follow symlink/junction loops.
- Native full-scan root discovery must report current-directory read failures instead of substituting `"."` when no user profile root is available. Native quick/full scan environment roots must reject parent-traversal segments before scan-root selection, so `USERPROFILE`, `HOME`, `TEMP`, `LOCALAPPDATA`, or `PROGRAMDATA` overrides cannot smuggle `..` into the scan plan. Checkpoint 1213 source-verifies this boundary.
- Skip quarantine, app caches, generated build outputs, and explicitly excluded directories unless configured otherwise.
- Bound memory use; stream large files where possible. Current native file scans stream full-file hashing and analyze a bounded 64 MiB sample with explicit sample-limit metadata. Guard pre-execution hashing also uses buffered streaming I/O. Local-core/guard compatibility YARA-style providers and local-core heuristic script/entropy checks read only bounded 1 MiB samples. Guard compatibility YARA rule files are capped at 256 KiB before parsing in both service and driver-verdict paths.
- Bound native scan concurrency. Native scan worker counts are clamped to documented 1-8 worker bounds, and platform parallelism-query failures fall back to one worker through an explicit error branch instead of a hidden default.
- Handle denied/locked/huge/unusual paths gracefully.
- Support cancellation and progress reporting.
- Real-time monitor duplicate suppression must not rely on size alone; the cache must include file modified time (or stronger content identity) so same-size payload replacements are scanned again.
- Checkpoint 1110 source-verifies Guard Service process hash caching is bound to non-following file length and modified-time identity so same-path executable replacements are rehashed instead of reusing stale known-bad hash evidence.

### Detection and action policy

Verdicts should include evidence, reason codes, confidence, and recommended action. Action policy must remain conservative:

- Clean/unknown/low-signal observations: no automatic action.
- Suspicious/probable findings: user review unless stronger independent evidence and explicit policy allow action.
- Confirmed signature/test-threat findings: may quarantine only when scan mode permits, path is not allowlisted/trusted, and quarantine succeeds safely.
- Strong probable-malware findings with independent evidence must be evaluated before stale local trust records such as known-good hashes, exact user approvals, or trusted-publisher allow decisions; in Lockdown they block pending review, and in less strict modes they are allowed only with monitoring/review.
- ML-only detections from development models must not auto-quarantine.
- User training labels may suppress only exact-hash false-positive/trusted-app review outcomes; suppression resolution must use the newest valid label for that hash so later confirmed-malicious/unsure labels can revoke stale suppression.
- Training-label newest-record selection must make both existing-label and no-existing-label branches explicit, so first-label behavior is not hidden behind optional defaulting.
- Known-good, known-bad, and user-approved exact-hash stores must accept only valid SHA-256 values. Malformed trust entries must fail closed.
- Checkpoint 913 requires exact-hash trust-store JSON schemas to be explicit. Guard known-bad, local app-control known-bad/known-good, and native known-good/known-bad stores reject unknown object or record fields instead of silently accepting policy-like evidence the trust engine ignores; existing descriptive metadata is allowed only through named schema fields.
- Guard-service known-bad, known-good, user-approved, and trusted caller-supplied hashes must use the same valid SHA-256 requirement.
- Local-core application-control input hashes must also be valid SHA-256 values before they can influence known-good, known-bad, or user-approved decisions. Malformed execution hash metadata is surfaced as suspicious/untrusted metadata, while stronger confirmed/probable malware signals still take precedence.
- Local-core application-control exact-hash lookups must make missing or malformed lookup hashes explicit. Known-good, known-bad, and user-approved lookup helpers may return no match for malformed input, but only through visible validation branches; policy routing must evaluate malformed input through the explicit malformed-hash path rather than hidden optional false defaults.

Allowlist decisions must not reduce protection through path-only trust for mutable files:

- Explicit hash entries may trust the same SHA-256 at any path.
- File, app, and executable entries must record the current file SHA-256 at creation time and require both the normalized path and hash to match at evaluation time.
- Legacy/path-only file, app, and executable entries must fail closed instead of trusting mutable paths.
- File, app, and executable allowlist creation must fail closed when the target cannot be hashed.
- File, app, and executable allowlist creation must also bound selected-file hash inputs before durable trust evidence is saved. Local-core allowlist hashing inspects the target without following links, rejects oversized metadata before open, and tracks actual bytes read while hashing so a replaced or growing file cannot force unbounded trust-entry hashing.
- Folder entries are path-scoped and should remain reserved for explicit user-controlled folder trust decisions.
- Exact-hash trust entries must be valid SHA-256 values. Malformed known-good, known-bad, user-approved, or allowlist hashes fail closed and must not create trust.
- Signature hash patterns must be valid SHA-256 values. Exact-hash signatures require full SHA-256 equality, and partial-hash signatures require at least a 32-character hexadecimal SHA-256 prefix; malformed hash signatures fail closed and should be rejected during definition compilation.
- Native hash-signature runtime matching must preserve that contract for active definitions and internally computed scan hashes. Invalid exact-hash patterns, invalid partial-hash patterns, or invalid actual SHA-256 values fail visibly instead of being reported as clean no-matches.
- Signature compiler source JSON must be size-bounded before parsing so source packs cannot force unbounded compiler memory use before validation.
- False-positive suppression and user approval hashes must also be valid SHA-256 values; malformed entries fail closed and must not suppress detections or approve execution.
- Threat-intel-generated signatures must not include unrecognized required-context markers. Imported indicators are compiled into native signatures whose matching behavior is determined by the declared signature type, normalized pattern, file-type constraints, confidence, and action policy.
- Threat-intel hash-line import must not manufacture empty hash fields. Blank and comment rows are skipped through an explicit helper branch, while non-empty malformed SHA-256 fields fail visibly with line context instead of becoming empty or no-op indicator evidence.
- Native rules must validate match counts, file types, import categories, entropy bounds, numeric thresholds, and short string/path needles before activation. A rule with zero required matches, impossible required matches, unsupported categories, invalid thresholds, or a single broad short string/path condition must fail closed during rule compilation.
- Native rule packs must declare the supported rule-pack format and a non-empty version before individual rules are activated.
- Native rule packs are detection inputs, not extensible policy blobs. `RulePack`, `NativeRule`, `RuleCondition`, and `RuleMatch` schemas must reject unknown fields so disabled/enabled flags, allow overrides, or policy-looking controls cannot be silently ignored while appearing present in local rule evidence.
- Threat-intel signature pack tooling must emit generated signatures without unrecognized required-context markers and include a self-hash over canonical pack JSON. Existing auxiliary packs without hashes remain a compatibility migration item until they can be regenerated and runtime verification can require hashes for every signature pack.
- Auxiliary native signature packs should not carry unrecognized prose required-context markers; hashed core packs must be regenerated rather than edited in place when context metadata changes.
- Native signature required-context values must be explicit recognized condition keys. Unknown context strings fail closed; the current hashed core pack has a narrow exact-string compatibility exception until it can be regenerated.
- ClamAV compatibility detections must not have empty detection evidence. Exit code 1 is treated as infected, and if scanner output does not include a parseable signature name, local-core uses `ClamAV compatibility detection` as the conservative evidence label.
- Native signature and rule pack loaders must load only regular pack files and must reject symlinks, junctions, or reparse redirection; sibling discovery should not follow linked pack entries.
- Non-empty native signature packs must include a valid `pack_sha256` self-hash matching the Rust canonical pack representation. Empty compatibility packs may remain hashless because they activate no signatures. This self-hash is an integrity check, not a remote publisher signature.
- Python indicator-pack validation is a definition-chain gate for generated and bundled `.zsig`/`.zrule` packs. It must reject unknown pack, item, and condition fields; non-object entries; missing explicit signature/rule lists; invalid self-hash syntax; empty rule conditions; and impossible condition counts before reporting validation success. The small-threat MVP verifier must include both the harmless threat-intel pack metadata smoke and validation of the bundled signature/rule packs, including expected-inventory checks for the shipped ransomware, infostealer, miner/PUP, persistence, and script-threat packs. The full-suite report validator must reject reports that omit that coverage, omit the generated bundled pack inventory report, or provide an inventory whose schema, counts, paths, byte sizes, expected pack rows, or validator-output evidence do not match the current repository.
- Non-empty native rule packs must include a valid `pack_sha256` self-hash matching the Rust canonical pack representation. Empty compatibility rule packs may remain hashless because they activate no rules. This self-hash is an integrity check, not a remote publisher signature.

## Protection claim evidence

- Synthetic non-driver protection evidence must not imply installed-driver, kernel, or pre-execution blocking. The small-threat MVP report validator must parse the generated `protection_self_test` report and require the synthetic fixture name, explicit no signed-driver/pre-execution disclaimer, non-driver policy/verdict successes, and `driver.communication_port_ok=false`, `driver.installed=false`, `driver.running=false`, `driver.pre_execution_blocking_available=false`, and `tests.unknown_unsigned_lockdown_blocked_before_launch=false`. ## Quarantine safety model

- Quarantine should move files rather than delete them.
- Quarantine records must store original path, normalized safe quarantine path, detection/evidence, timestamp, hash, and size where available. The copy fallback verifies the quarantined payload hash before removing the original.
- Restore must require explicit confirmation.
- Restore, permanent-delete, and allowlist-removal IPC commands must receive an explicit `confirmed=true` field at the command boundary; missing confirmation is a visible error and must not be defaulted through optional booleans.
- Manual quarantine, restore, and permanent-delete actions must be single-flight at the Flutter controller boundary so duplicate clicks or cross-action races cannot send overlapping quarantine mutation IPC to local-core.
- Restore must reject path traversal and unsafe destinations.
- Restore must verify quarantined payload size and SHA-256 before moving the payload back to the original path, so tampered quarantine files cannot be restored.
- Restore must not overwrite existing files without explicit separate handling.
- Permanent delete must require explicit confirmation.
- Scan result rows must not expose destructive delete controls. Deletion is available only for quarantine records after the local core has isolated the payload and the user confirms the Quarantine-screen action.
- After a restore or permanent-delete action succeeds in the local core, the UI must immediately mark the local quarantine row restored or deleted before refreshing the list. A refresh failure may show an error, but it must not leave the already-acted record enabled as though it were still quarantined.
- Local quarantine environment roots are trust boundaries. `AVORAX_QUARANTINE_DIR`, `ZENTOR_QUARANTINE_DIR`, ProgramData-derived roots, and HOME-derived roots must reject NUL and parent-traversal segments before `PathBuf` construction and absolute-local path validation. Checkpoint 1208 source-verifies this for local quarantine store construction.
- Local-core quarantine ACL hardening must resolve `icacls.exe` through a checked local System32 path with non-following metadata before launch. Local-core System32 discovery must fail visibly when `SystemRoot` and `WINDIR` are missing, empty, remote, relative, contain parent traversal, or otherwise are not local drive paths; it must not silently substitute `C:\Windows` or discard environment-read failures through optional defaults. ACL helper stdout is discarded, stderr is read through a bounded pipe reader while the helper runs, command execution has a finite timeout with visible kill/reap cleanup diagnostics, and bounded stderr is the only helper output allowed into quarantine errors. Checkpoint 1202 source-verifies parent-traversal rejection before local-core `sc.exe` or `icacls.exe` launch paths are built; checkpoint 1297 source-verifies local ACL timeout cleanup diagnostics.
- Corrupted metadata should produce an actionable error and should not cause data loss.

## Settings and configuration model

Settings are local user preferences, not authoritative security proof. They must be validated before persistence:

- Real-time protection enabled/disabled.
- Ransomware protection enabled/disabled.
- Scan exclusions.
- Protected folders for ransomware monitoring.
- Trusted backup/sync process allowlists for ransomware-policy suppression.
- Allowlist entries.
- Scan sensitivity.
- Automatic update preference.
- Notifications.
- Cloud/developer overrides.

Unsafe broad allowlist/exclusion/protected-root values such as drive roots, system directories, `/`, `/usr`, `/bin`, `/sbin`, and `/etc` should be rejected or require exceptional explicit handling. Allowlist path validation must normalize slash/backslash spelling before broad-root checks and reject exact single-letter drive roots, any-drive `Windows`, `Program Files`, `Program Files (x86)`, `ProgramData`, and `Users` roots, plus Unix `/`, `/System`, `/usr`, `/bin`, `/sbin`, and `/etc` roots. Normal user-owned subfolders such as `C:\Users\Example\Tools` may still be allowlisted only through the relevant hash/path entry rules. Current ransomware protected-root IPC rejects broad root-style protected folders before persistence.

Allowlist path matching is trust-boundary logic. Local allowlist path validation and folder/file matching must collapse `.` and `..` path segments before checking blocked roots or comparing candidate scan paths, so traversal spellings cannot escape a folder allowlist or hide an unsafe root-shaped trust entry.

Guard mode and ransomware guard policy config writes must be staged before activation and must reject symlink, junction, or reparse-point config files and parent directories.

Guard-service guard mode config is persisted privilege-boundary policy and must be size-bounded before parsing. The guard service rejects oversized `guard_mode.json` data through a bounded reader before JSON or legacy text mode parsing, so local file tampering cannot force unbounded privileged config parsing work.

Checkpoint 909 treats JSON guard mode config as strict privilege-boundary policy evidence. The guard service parses `guard_mode.json` through a deny-unknown-fields schema, validates bounded mode/source text, and rejects empty/NUL/oversized source or mode evidence instead of accepting JSON fields that appear to configure controls the Guard will ignore.

Guard-service guard mode config paths must be inspected without following filesystem links before defaulting or bounded reads. Missing shared config still defaults safely, but present symlinks, Windows reparse points, directories, and other non-regular config paths fail visibly instead of being followed or treated as absent.

Guard-service protection-mode environment overrides are privileged policy input. Missing override variables may fall through to shared config, but present unsupported or non-Unicode override values must fail visibly instead of being ignored and replaced by config/default mode.

Cross-process runtime sentinels such as scan cancellation tokens must not follow symlinks, junctions, or reparse points and should be staged before activation where practical.

Current native file scans stream full-file hashing and analyze bounded samples for compatibility and heuristic engines. Local-core YARA-style rule inspection, heuristic script/entropy checks, the ClamAV compatibility local-signature precheck, and guard-service compatibility YARA checks read only bounded 1 MiB samples; guard-service compatibility YARA rule files are capped at 256 KiB before parsing. Checkpoint 1283 source-verifies local heuristic, local YARA, Guard YARA, driver-IPC YARA, and local static-feature sample prefixes use explicit chunk limits; checkpoint 1284 does the same for the local ClamAV/EICAR signature precheck. SHA-256 helpers stream the full file so hash semantics are exact without unbounded memory use.

Guard-service compatibility YARA rule file paths must be inspected without following links before bounded reads in both service scan and driver-verdict paths. Missing rule files may disable the compatibility rule signal, but present symlinks, Windows reparse points, directories, or otherwise non-regular rule paths fail visibly before parsing.

Guard-service compatibility YARA match ranking must not hide first-match behavior behind optional defaults. The first matched rule is accepted through an explicit empty-best branch, and later matches replace it only when their confidence rank is higher.

Scan reports must not silently swallow scan root, file-walk, metadata, file read, or native-engine scan errors. These failures are counted as skipped, included in a bounded `scan_errors` list, and described as not reported clean. The local core, native scan-summary helper, and file walkers bound each scan-error detail, normalize NUL bytes before report serialization, and record omission notices when additional scan-error details are dropped at list caps. The UI must present completed-with-errors diagnostics before any "no threats found" empty state.

Local ClamAV compatibility scanning treats external scanner output as untrusted evidence. Scanner stdout/stderr are drained through bounded-retention pipe readers, each stream is capped and marked when truncated, and the child process is timed out so noisy or stuck scanner integrations cannot flood scan summaries or hold a scan indefinitely without a visible scanner error. Checkpoint 1281 source-verifies the local ClamAV readers drain stdout/stderr while retaining only bounded diagnostic bytes.

Local ClamAV compatibility errors are detection evidence. The local EICAR signature precheck returns contextual read/path errors instead of defaulting failures to no-match, and timeout cleanup reports failed process reaping so child-process cleanup failures cannot be hidden behind a generic timeout.

Local ClamAV scanner discovery errors are operational evidence. A configured scanner path that is empty, invalid, NUL-containing, parent-traversing, relative, non-local, linked, reparse-backed, non-file, or uninspectable fails visibly instead of falling through to bundled scanners. Bundled scanner discovery depends on the running executable location, and failures to inspect that location or unsafe bundled scanner/database candidates are reported as scanner discovery errors rather than silently becoming "not installed."

Local UI actions that call the local core for quarantine, restore, delete, allowlist, or detection labeling must preserve local-core error text. The UI may add context, but it must not collapse validation, integrity, path-safety, permission, or IPC failures into indistinguishable generic false results. Quarantine action failures must be quarantine/error events, quarantine restore/delete/request/success mutations must be quarantine warning history, and allowlist trust mutations/failures must be protection warning/error history.

Flutter temp-file cleanup after local event export or in-app update package download failures must be fail-visible. Cleanup failures must include the original operation error and the cleanup error rather than being silently swallowed.

Local event export temporary cleanup is a failure-path trust boundary. Failed exports remove temporary export files only after non-following type checks confirm the target is a regular file; link, directory, and other unsafe cleanup targets fail visibly alongside the original export error.

The Flutter in-app updater treats downloaded `.aup` packages as untrusted until verified. Downloads must be staged to temporary package files, hashed before activation, and activated only into non-linked regular cache destinations. Remote package bytes are streamed into the reserved temporary file while content-length and accumulated byte limits are enforced; full package bodies must not be buffered in memory before staging. Linked cache directories, linked package paths, non-regular destination objects, colliding temporary package paths, and linked or non-regular local `file:` package sources fail closed.

The native update service must create staging, install, rollback, and payload destination directories only through checked directory creation. Existing symlink/reparse ancestors are rejected before and after `create_dir_all`, so update apply or rollback cannot create files through a linked directory chain.

Native update payload replacement and rollback file restoration must use staged sibling files rather than direct target overwrites. Staged files are written, flushed, synced, and then activated with rename after a final linked/reparse target check. Staged temp names include process/time/attempt components so stale regular temp files do not block later updates; inability to read system time fails visible instead of defaulting temp-name entropy to zero; linked/reparse temp candidates and non-`NotFound` metadata errors fail closed. If activation fails after staging, the `.avorax-part` file is removed before returning the original error. This reduces partial-file risk during update apply or rollback failures.

Native update package payload extraction follows the same staging rules: extraction directories use checked creation, temporary payload files use bounded process/time/attempt `.avorax-part` names, and only `NotFound` metadata results are accepted for temp allocation. System-time failures fail visible instead of producing low-entropy extraction temp names. Stale regular temp files must not block future update extraction, while linked/reparse candidates fail closed.

Native update report logs are operational evidence and must not be written through unsafe paths. Update log names are restricted to safe basenames, update log directories use checked directory creation, and report contents are written through staged byte activation with linked/reparse checks.

Update CLI status files are operational evidence. Successful update-service CLI commands must not exit successfully after losing status-write evidence; status-write failures are surfaced on stderr and cause a nonzero exit on otherwise successful commands. Failed commands also print any status-write failure instead of discarding it.

Native update package paths are untrusted inputs. Before manifest parsing, signature reads, package hashing, payload verification, or extraction, the update service requires the package path to reject NUL bytes and parent-traversal components, then prove the target is a regular non-linked/non-reparse non-empty `.aup` file within the configured package-size ceiling. Package file opening repeats the path safety check after `File::open` and compares pre-open path metadata, opened-handle metadata, and post-open path metadata before reading from the handle, so obvious link/reparse or replacement races fail visibly before package bytes are trusted. Package hashing must also use this checked opener; direct path-hash helpers that reopen arbitrary paths are not part of the update package verifier. ZIP archive entry counts are capped before manifest/signature entry reads, payload hashing, or extraction loops continue. Manifest and signature ZIP entries must be streamed with actual-byte accounting before UTF-8, Ed25519 signature decoding, or manifest JSON parsing; checkpoint 1282 source-verifies this boundary. Payload hash manifests reject duplicate normalized paths before the archive is opened, and payload archive validation rejects duplicate normalized payload paths so one signed/hash-checked name cannot map to multiple entries. Payload extraction revalidates the canonical extraction destination as an existing non-linked directory before writing staged payload files. This metadata check reduces path-race ambiguity but does not replace signed manifest, package hash, and payload hash verification. CLI `--verify` and `--apply` must not follow package symlinks or accept non-package file extensions.

Verified update payloads are copied from staging into the installed layout through staged file activation. The update destination copier validates the supplied source and destination, canonicalizes them, then revalidates the canonical roots as existing non-linked directories before walking payload entries or deriving copy targets. This reduces source/destination replacement ambiguity during update application; it does not remove the need for elevated service isolation and real signed-package apply/rollback validation on a provisioned Windows host.

Rollback directory restores use the same staged copy posture. Snapshot and destination roots are canonicalized and then revalidated as existing non-linked directories before rollback file walking or staged copy starts, reducing ambiguity if a snapshot or restore destination changes between the first guard and canonical path use.

Rollback snapshot discovery treats the rollback root, snapshot entries, and snapshot directory names as untrusted filesystem input. Latest-snapshot selection rejects linked/reparse rollback roots, fails visibly on linked/reparse snapshot entries before candidate selection, and considers only directories whose names pass the same safe snapshot-name policy used for snapshot creation.

Rollback snapshots are complete-or-fail artifacts. Snapshot creation and restore require the app executable, core service executable, guard service executable, and engine directory to exist with the expected type and without symlink/reparse redirection. Recursive snapshot restore copying must also reject linked/reparse snapshot entries before deriving destination paths or staged copies. Missing or partial rollback contents fail closed instead of being skipped.

Update service stop/start operations are part of the update trust boundary. Windows service-control command failures must be returned as update failures, not ignored. Update Service service-control helpers must launch `sc.exe` through a checked local System32 path rather than ambient `PATH` lookup, must fail visibly when `SystemRoot`/`WINDIR` are missing, unsafe, or contain parent traversal instead of substituting `C:\Windows`, and must reject linked/reparse/non-regular `sc.exe` candidates before launch. Checkpoint 1206 source-verifies parent-traversal rejection before the service-control helper builds the `sc.exe` launch path, and checkpoint 1289 source-verifies start/stop commands close stdin, drain bounded stdout/stderr pipes, time out hung service-control commands after 30 seconds, report timeout kill/reap/read failures visibly, and avoid production `.status()` waits. Non-Windows builds must fail visibly for update-service stop/start operations instead of returning a successful no-op. If payload replacement succeeds but services fail to restart, the update must be treated as failed: restore the rollback snapshot where possible, attempt to restart the restored services, and write a failed update report containing restart and rollback status.

Update-service ProgramData roots are update supply-chain evidence roots. `AVORAX_DATA_DIR`, ProgramData-derived roots, and HOME-derived roots must reject NUL and parent-traversal segments before `PathBuf` construction and absolute-local path validation, because the same root feeds update logs, staging, and rollback directories. Checkpoint 1210 source-verifies this shared update-service root parser.

Update Service `--service` mode is Windows-service behavior, not a portable daemon shim. Non-Windows builds must fail visibly instead of entering a blocking placeholder loop, so CLI status/error evidence can report unsupported service mode rather than hanging without proof of protection.

Failed update report writes are also operational evidence. When writing the failed update report fails after an apply or service-restart failure, the update service attaches that report-write failure to the returned error context instead of ignoring it, while preserving rollback and rollback-restart failures as the higher-priority returned errors.

Archive analysis is evidence-only and does not extract payloads. ZIP/JAR/APK/XPI/VSIX/NUPKG/APPX/MSIX/APPXBUNDLE/MSIXBUNDLE path checks treat parent traversal, POSIX absolute paths, Windows drive-root paths, root-backslash paths, and UNC-style paths as unsafe archive paths. Invalid ZIP-framed entry metadata fails the analyzer, while truncated entry bodies are marked as limited analysis rather than clean evidence. Local-core must surface otherwise non-surfaced `archive_content_scan_limited` evidence as skipped-file scan errors and `completedWithErrors`, so oversized, unsupported, or otherwise unscanned archive content cannot be reported as clean success.

ZIP executable-entry evidence must not manufacture empty suffixes. Entry suffix parsing uses an explicit empty-suffix branch before executable/script suffix matching so archive evidence is explainable and source-testable.

## Logging and privacy

Avorax logs structured security/protection events and operational errors without leaking file contents. Local events include type, timestamp, message, optional details, category, and severity; user-initiated export serializes those fields as JSON.

In-app notifications are derived from real local events. The shell may surface recent warning/error and important scan, update, or quarantine events, but Avorax must not claim OS toast notifications until a Windows notification workflow is implemented and verified.

In-app notification text is display-bounded. The shell collapses event message/details whitespace, caps the banner text, and renders a single line; full details remain available in Logs/export instead of expanding the global shell layout.

Allowed logging:

- Event type, timestamp, status, path metadata where relevant, hashes, counts, reason codes, scan IDs, update IDs, and actionable errors.

Avoid logging:

- File contents.
- Secrets, tokens, private keys, or update signing secrets.
- Excessive user document data beyond paths necessary for local event history.

Exports should be user-initiated and should clearly identify what data is included.

- User-mode monitoring cannot prevent all execution or file writes before they happen.
- Local process and behavior monitor health must separate platform capability from active runtime status. In this build they report `notActive` with an explicit no-loop reason until a local-core polling/observation loop is implemented and runtime-tested.
- Flutter health parsing and Protection UI must preserve local process and behavior monitor status, capability, and no-loop reasons instead of dropping those fields or inferring active monitoring from platform capability.
- The bundled native ML model is currently a development model and cannot support production AI claims or automatic quarantine by itself.
- Cloud reputation is optional/disabled unless a real backend is configured; disabled/unavailable provider states must be represented honestly and must not contribute evidence or imply cloud coverage.
- The local reputation provider is currently an explicit no-op with status `unavailable` and a health reason of `no local or cloud reputation backend is configured`; it must not produce `ThreatResult` evidence or imply local/global reputation coverage until a real bounded reputation backend exists.
- Flutter health parsing and local UI surfaces must preserve reputation provider status and reason evidence. A disabled or unavailable reputation provider should remain visible as unavailable with its no-backend reason instead of disappearing behind generic engine status.
- ProgramData root paths and resolution errors are local health evidence. Flutter health parsing and local UI surfaces must preserve `program_data_dir` and `program_data_dir_error` evidence, clear stale ProgramData roots when later health responses omit them or fail, and keep invalid/missing ProgramData roots visible instead of collapsing them into generic engine failure.
- Local-core IPC boundary evidence must remain explicit. Flutter health parsing and Settings UI must preserve the local-core `ipc` mode and nullable `network_exposed` evidence so the app reports local stdio IPC only when the core says so and reports unknown rather than claiming "not network exposed" from missing or malformed protocol data.
- Native engine failure diagnostics are native-engine evidence. Flutter must preserve `native_error` separately from aggregate `last_error` and self-test detail text, clear stale native-engine error evidence after recovery, and display it near native-engine status surfaces.
- Aggregate local-core `last_error` is health evidence, not sticky status. Flutter must preserve bounded `last_error` diagnostics when local-core reports them, but must clear stale aggregate engine errors when later health responses omit `last_error` after recovery.
- Native and AI self-test booleans are health evidence. Flutter must preserve nullable `native_self_test` and `ai_self_test` pass/fail/unknown states separately from error text, clear stale self-test evidence after health failure, and display unknown rather than inferring success from absent errors.
- Native asset directories are operational health evidence. Flutter health parsing and Settings UI must preserve signature, rule, ML, trust, and config directory paths from local-core health so pack discovery and deployment issues can be diagnosed without inferring path state from aggregate engine status alone.
- Install and engine root discovery paths are operational health evidence. Flutter health parsing and Settings UI must preserve local-core `install_path`, `engine_directory`, and bounded `engine_paths_checked` evidence, and must clear stale root evidence when later health checks omit it or fail.
- Core Service and Guard Service status errors are service-boundary evidence. Flutter must preserve `core_service_status_error` and `guard_status_error` separately from aggregate last-error text, clear stale service errors after recovery, and display them near the service status surfaces.
- Local AI compatibility status is separate from native ML status. Flutter must preserve top-level `ai_status` from local-core health and must not render the Protection "Local AI" checklist from `native_ml_status`, because those represent different engine surfaces and readiness claims.
- Some Windows service and update tests require elevation.
- Driver validation requires Microsoft signing and a real self-test report.

- Use harmless synthetic fixtures only.

### Guard metadata trust boundary

Guard pre-execution decisions must not trust caller-provided publisher, signature, or hash metadata unless that metadata names a trusted verifier source such as the Avorax kernel driver, Avorax guard service, Windows Code Integrity, or Windows WinTrust. When a file is readable, the guard computes its own SHA-256 with streaming I/O and prefers that over supplied metadata. Supplied hashes are accepted only as a fallback for unreadable race-window files and only with trusted verifier provenance. The guard caches one native engine instance behind a mutex for repeated pre-execution verdicts; this improves latency without changing the fail-open policy for critical OS/Avorax runtime paths.

Guard fail-open path exemptions must be exact-root decisions, not substring matches. Windows system exemptions apply only below `SystemRoot`/`WINDIR` configured Windows system directories; if those roots are absent, no conventional `C:\Windows` system exemption is inferred. Quarantine exemptions apply only below configured or conventional product quarantine roots, and service executable exemptions apply only to the running executable or known product install roots. Lookalike user paths such as `C:\Users\Public\Windows\System32\...` or `C:\ProgramDataX\Avorax\Quarantine\...` remain unknown and must follow the selected protection mode.

Guard fail-open runtime roots are environment-derived trust inputs. Relative, NUL-containing, or parent-traversing `SystemRoot`, `WINDIR`, ProgramData, ProgramFiles, quarantine, or HOME roots must be ignored instead of normalized into trusted fail-open roots; checkpoint 1220 source-verifies this for driver IPC root candidate enumeration.

Guard fail-open prefix checks must make non-descendant cases explicit. Exact-root, missing-prefix, and descendant decisions should be visible branches rather than optional false defaults hidden inside string-prefix helpers.

Guard fail-open comparisons must collapse `.` and `..` path segments before checking system, product, or quarantine descendants. A path under a fail-open root followed by `..` must not remain fail-open trusted after it escapes that root, especially in Lockdown where unknown paths should block or require approval.

Local-core application-control passthrough paths follow the same exact-root rule. System passthrough applies only below `SystemRoot`/`WINDIR` configured Windows system directories or Unix system roots; if those Windows roots are absent, no conventional `C:\Windows` system passthrough is inferred. Product passthrough applies only below configured or conventional quarantine roots, the running executable, or service executables under known product install roots. User-controlled lookalikes such as `C:\Users\Public\Windows\System32\...`, `C:\ProgramDataX\Avorax\Quarantine\...`, or `C:\Users\Public\zentor_local_core.exe` must not become `SystemTrusted` merely by containing trusted-looking path fragments. Passthrough environment roots must also reject NUL and parent-traversal segments before `PathBuf` construction and exact-root trust decisions; checkpoint 1211 source-verifies this boundary.

Local-core application-control passthrough prefix checks must make non-descendant cases explicit. Exact-root, missing-prefix, and descendant decisions should be visible branches rather than optional false defaults hidden inside string-prefix helpers.

Local-core application-control passthrough comparisons must collapse `.` and `..` path segments before checking product, quarantine, or system descendants. A path under a trusted root followed by `..` must not remain trusted after it escapes that root, and traversal-derived broad allowlist roots must be treated as dangerous before policy storage.

### Ransomware protected roots

Ransomware evaluation supports explicit protected roots and trusted process allowlists. If protected roots are configured, only activity inside those roots contributes to modification thresholds. Trusted backup/sync processes can be suppressed by exact normalized path for ordinary mass-modification signals, but trusted-process suppression must not hide critical ransom-note or backup-tamper signals because trusted software can be compromised. Flutter settings persist these lists, local core IPC writes the shared policy config, protected roots are included in user-mode watch planning, and local event history records protection/ransomware settings changes with category/severity metadata. Tests use harmless temporary/path-only fixtures and do not encrypt or damage files.

Protected-root and trusted-process comparisons are local policy boundaries. Local ransomware evaluation must collapse `.` and `..` path segments before matching event paths against configured protected roots or trusted-process allowlists, so traversal spellings cannot be counted as protected activity outside the root and lexically equivalent trusted process paths are compared consistently.

Settings mutations that affect trust endpoints or local policy state must report write failures honestly. Developer cloud overrides, protection profile changes, ransomware settings, and configuration reset controls must not show success UI unless persistence and required local-core writes complete.

Protection profile, ransomware guard, and scheduled quick scan settings actions must be single-flight at the Flutter controller boundary so duplicate clicks or cross-setting races cannot overlap local-core policy writes, app-config persistence, or scheduled-timer reconfiguration. The Settings UI must expose that boundary by disabling the shared profile, ransomware policy text fields, ransomware-save, and scheduled-scan setting controls while a security settings write is in flight.

The Flutter configuration repository is itself a policy persistence boundary. Saving or resetting persisted policy must check the `SharedPreferences` boolean acknowledgement and fail visibly if storage rejects the write or removal; callers may only report success after the repository accepts the persisted policy operation.

Protection profile changes are Guard-policy mutation boundaries. Flutter controller profile changes must reject direct `Off` selection through the profile dropdown and require the Stop Protection flow instead. Non-Off profile changes must fail closed unless called with explicit confirmation because they alter Guard behavior and local protection policy. The Settings UI may pass `confirmed: true` only after the profile-change dialog is accepted. Confirmed saves must write the shared local-core Guard mode before persisting Flutter app config; local-core Guard mode rejection preserves current settings, and post-core persistence failure attempts best-effort Guard-mode rollback with visible diagnostics.

The outer elevation launcher must use a checked local WindowsPowerShell executable path and UTF-16LE `-EncodedCommand`; the inner elevated Start-Process wrapper must point to that same checked PowerShell path.

Before self-test, scan IPC, or cancel IPC starts a helper executable, Flutter must use an absolute launch path, reject Windows UNC/device paths, and re-probe the path as a regular non-linked file so environment overrides or stale discovery results cannot redirect execution to ambiguous targets.

Local-core environment roots are runtime trust boundaries. Engine asset roots, ProgramData/runtime roots, and shared config roots supplied through environment variables must reject NUL and parent-traversal segments before absolute-local path validation, so an override cannot smuggle `..` through an otherwise absolute path. Checkpoint 1207 source-verifies this for engine asset, ProgramData, and shared-config env-root entrypoints.

Confirmed service/report actions must not rely on local-core returning a clean status string. Guard-service process stop actions are part of the auto-action evidence boundary. If a PID is supplied, the stop command must launch successfully, return a successful status, and surface bounded stderr on failure before quarantine proceeds. If no PID is supplied, Avorax may quarantine the matched file but must not claim the process was stopped. Normal `.aup` app payloads target the install root and therefore must not become a bypass around specialized update sections. The verifier, builder, and applier must reject `payload/app` entries whose first install-root child is `engine`, `docs`, `tools`, `driver`, `driver-tools`, or `migrations`, and must reject direct managed service/updater executable names under `app`. Those surfaces require their explicit payload section, MSI/EXE packaging, or a separate workflow with its own verifier and rollback contract.

Normal `.aup` engine payloads are a runtime-pack update surface, not a generic engine filesystem replacement surface. The update verifier, builder, and applier must accept or activate only files under `engine/signatures`, `engine/rules`, `engine/ml`, and `engine/trust`; `native_engine_assets` is a parent inventory declaration and must not authorize unknown engine subdirectories, direct engine-root files, `engine/config`, `engine/test_corpus`, or `engine/threat_intel`. The applier must not copy `staging/engine` wholesale; it must enumerate supported component directories, reject unknown or duplicate normalized component names, and copy each supported component to its canonical destination. Config, test-corpus, threat-intel, native binary, or other engine-layout changes require MSI/EXE packaging or a separate explicit workflow with its own verifier, manifest contract, rollback rules, and runtime tests.

Normal `.aup` service payloads are also an explicit file allowlist. The verifier and applier must accept or activate only direct `avorax_core_service.exe` and `avorax_guard_service.exe` payload files; the applier must not copy `staging/services` wholesale, must reject nested service directories or unsupported service filenames, and must use the bounded staged-file activation path for each service binary.

Normal `.aup` docs payloads are documentation-only. The verifier, builder, and applier must accept or activate only regular Markdown files under `docs`; scripts, binaries, bare `.md` names, linked/reparse-backed entries, and other unsupported docs paths must fail visibly. The applier must not copy `staging/docs` wholesale and must stage accepted docs through the bounded file activation path.

The bundled Rust update manifest signer treats the manifest path as untrusted local input. It validates absolute local path shape, non-linked ancestors, non-link manifest metadata, and declared size before signing, then streams manifest bytes through an actual-byte cap before computing the signature so a replaced or growing file cannot force an unbounded read.

In-app update package caching is untrusted input handling. The Flutter update client must validate feed package names, reject unsafe package/cache paths, bound download sizes, reserve temporary cache files with exclusive creation before writes, stream local and downloaded package bytes into those reserved temp files with regular-file rechecks and byte-count limits, hash the staged package before activation, and fail visibly on unsafe temp path or cleanup races.

Cached update package activation must preserve rollback evidence for the local cache. When a cached `.aup` already exists, the Flutter client must move it to a checked temporary backup before activating a newly staged package, restore that backup if activation fails, and report combined activation/restore failures instead of deleting the existing cache entry first.

Windows protection self-test is driver validation evidence, not a normal user workflow. `tools/windows/zentor-protection-selftest.ps1` must use a script-relative repository root, require an explicit checked Cargo executable through `-CargoPath` or `CARGO`, validate build/install/self-test scripts before invocation, launch child build/install/self-test commands through bounded temp-file diagnostics, keep its report path under the repository, and pass the checked built Guard Service executable to the minifilter self-test. Ambient Cargo lookup is not trusted release evidence.

Live driver remediation is an elevated development-VM recovery boundary. `tools/windows/avorax-fix-driver-live.ps1` and the explicit TESTSIGNING helper must keep TESTSIGNING and SYSTEM post-reboot task creation behind confirmation switches, resolve administrative tools from checked local System32 paths without silent `C:\Windows` fallback, write remediation reports and generated post-reboot scripts through atomic staging, capture bounded `bcdedit`/`schtasks` and generated post-reboot diagnostics, and ensure generated SYSTEM scripts revalidate embedded tool and install-helper paths as regular non-reparse files before launch. These reports are development evidence only and must not be used to claim production pre-execution protection without the signed-driver workflow.

Live driver remediation logs are operational evidence. The helper must not append through unchecked log paths or let command output grow without bounds. Live command captures and final catch diagnostics must record exit codes where available, bound stdout/stderr or exception text before logging/reporting, read existing log content through bounded handle/tail reads before append, and atomically activate log updates; generated post-reboot reports must run install/load/filter/service probes through bounded temp-file diagnostic subprocesses before JSON serialization. Checkpoint 1299 source-verifies existing `latest.log` reads no longer use `ReadAllText`.

Windows driver build/setup evidence is a release-chain trust boundary. Driver setup/build scripts must not discover MSBuild, signing tools, or INF tools through ambient PATH lookup. Tool discovery is limited to checked local Program Files and Windows Kits candidates, setup/build reports must stay under the repository validation output tree, report files must be atomically staged, delegating driver setup wrappers must validate shared setup scripts as checked repository files before launch, and project/tool/linker/object/artifact paths must be validated as regular non-reparse files before they become release evidence. Driver setup reports consumed by build/sign scripts must be parsed as bounded regular-file JSON through byte-limited handle reads before any tool path from the report is trusted. Shared Windows helper JSON reads must follow the same bounded-handle rule. Driver setup tool enumeration failures and unsafe existing-candidate validation failures must be bounded `tool_discovery_errors` plus setup errors instead of missing-tool assumptions, and driver object, build-artifact, and signing-target enumeration failures must fail visibly instead of becoming empty artifact evidence. Secure Boot and TESTSIGNING setup evidence must distinguish disabled from unverifiable: `Confirm-SecureBootUEFI` failures are bounded `secure_boot_probe_error` report fields and `bcdedit /enum` failures are bounded `test_signing_probe_error` report fields, not silent disabled-state defaults.

Windows driver test signing is development evidence only. Test-certificate creation and test-driver signing scripts must keep certificate/report outputs under the repository validation tree, require explicit confirmation before creating a user-store development certificate, validate certificate names and thumbprints as bounded tokens, sign only checked regular non-reparse driver artifacts under the repository, use checked `signtool.exe` evidence from setup discovery, capture bounded per-target `signtool` sign/verify `command_diagnostics`, write reports atomically, and use HTTPS timestamping. These workflows do not establish production kernel-driver signing; production signing remains a Microsoft Hardware Dev Center process.

Signing wrappers are launch boundaries too. A wrapper that delegates to a shared driver signing script must validate the delegated script, report path, and build-output directory as regular non-reparse repository paths before launch instead of relying only on downstream validation.

Windows driver validation reports are development evidence only. Test-driver install, uninstall, and self-test scripts must keep reports and setup reports under the repository validation tree, validate scripts and driver artifacts as regular non-reparse files before trust, write reports atomically, and require explicit confirmation before any install or uninstall action. Install scripts must capture bounded `command_diagnostics` for TESTSIGNING probes, driver install, driver load/start, and filter/service status queries, and must not report success unless the expected filter or service state is verified. Uninstall scripts must capture bounded `command_diagnostics` for unload/stop/delete commands and must not report success when unload/delete commands fail. The minifilter self-test stages command input atomically, rejects Guard Service executables outside the repository before process launch, reads Guard subprocess stdout/stderr through bounded captures, and parses both the Guard event and nested self-test report as bounded JSON before evidence is trusted; process-guard self-test reports must capture bounded service-query `command_diagnostics`. Driver self-test command failures must be bounded report evidence.

Protection self-test wrappers that can install test drivers must require an explicit install confirmation separate from selecting the self-test workflow. A build/test wrapper must not forward `-ConfirmDriverInstall` to driver installers unless its own `-ConfirmDriverInstall` switch was supplied.

Windows driver log collection is diagnostic evidence, not proof of protection. Log collector output paths must stay under the repository validation tree, be written atomically, and bound command/event-log text before it becomes report content. Missing services, failed command queries, and event-log read errors must appear in the report as explicit evidence instead of being hidden or represented as clean driver health.

ML evaluator reports are release evidence. Static and native ML evaluator metadata/model/fixture inputs must be bounded regular non-reparse files, report directories must be checked non-linked directories, existing report outputs must be regular non-linked files, and report writes must use exclusive UUID temporary files with atomic replacement. Development overrides may make the command exit successfully for evidence collection, but the report must keep `ok: false` and `status: development_blocked`.

Native ML fixture generation is development evidence too. `ml_native/build_features.py` must validate generated rows against the native feature schema and evaluator row rules, require positive and negative labels, reject linked/reparse output paths, and write JSONL through exclusive temporary files plus atomic replacement instead of direct destination writes. Native ML build, train, and export tools must share the same checked output helper for JSON/JSONL/model/asset writes so candidate `.zmodel` and exported metadata assets are not activated through weaker direct-write paths.

Static ML development outputs follow the same release-evidence rule. `ml/static_ml_schema.py` must reject linked/reparse inputs and unsafe outputs, use exclusive UUID temporary files for JSON, JSONL, and ONNX byte outputs, fsync temporary files before activation, and atomically replace final outputs. `ml/export_onnx.py` must write serialized model bytes through that helper instead of using direct `onnx.save` to the final or a weak temp path.

Performance benchmark reports are release-readiness evidence, even when they contain blocked metrics. `tools/perf/avorax-benchmark.py` must validate repository-contained output paths, reject linked/reparse report directories and files, write through exclusive UUID temporary files, fsync report JSON before activation, create harmless synthetic corpus files through exclusive fsynced writes, copy synthetic update fixtures through checked regular-file input opens and exclusive fsynced target creation, preserve both original copy and partial-cleanup errors when cleanup also fails, bound benchmark error diagnostics before report persistence, drain subprocess output through bounded capture, preserve timeout kill/reap cleanup failures, and keep non-passing metric statuses visible instead of reporting fake success.

Threat-intel generated JSON and JSONL packs are detection inputs. Shared `tools/zentor_intel` helpers must reject symlink/reparse inputs and output targets, create output parents only as regular non-linked directories, write through exclusive UUID temporary files, fsync before activation, and atomically replace outputs without deleting existing pack evidence first. Checkpoint 1293 source-verifies the real-world detection-pack wrapper reports bounded timeout kill/reap cleanup failures instead of hiding child cleanup failures.

Native threat-intel records are detection-source evidence, not extensible policy blobs. `ThreatIntelIndicator` and `ThreatIntelSource` schemas must reject unknown fields so allow overrides, enabled flags, or importer-only controls cannot be silently ignored while appearing present in compiled intelligence evidence.

Temporary output cleanup is part of release evidence integrity. Static ML, native ML, threat-intel, benchmark, and evaluator report helpers may accept the expected race where their own temporary file has already disappeared, but they must not use broad or empty cleanup handlers; unexpected cleanup-target mismatches must fail visibly with contextual diagnostics.
